Overview ¶

Common Items ¶

Auth ¶

Anonymous requests to the API are disallowed, so developers calling the API must provide some sort of key with each request (if you’re reading this document, you should already have this; if not, please contact us directly).

We issue two kinds of keys:

  • URL-based: when issuing an HTTP request, the client includes the key in a URL parameter called key (this is how e.g. Google Maps works). This method is simple, but because the key is passed directly in the request, it’s only suitable for clients that are running on trusted devices, such as a server you control, or a researcher’s workstation.

  • HMAC-based: the client signs an HTTP request with its secret key and passes the signature in a request header. Then, the server signs the same request using the same secret key, and verifies that the signature matches the one in the request header. This method is more complicated, but it makes it more difficult (though not impossible) to extract the key from clients running on untrusted devices.

For a Python example of how to use both auth schemes, see the examples folder.

HMAC Auth: Detailed Explanation ¶

To use HMAC-based auth, the following HTTP headers are required:

  • X-Kaiterra-Client (string): Contains your client’s unique ID that was issued to you by Kaiterra.

  • X-Kaiterra-Time (hex number): The current time, in UTC, expressed as Unix time (seconds since the epoch) and base 16 encoded. Leading zeroes should be dropped. Do not pass negative numbers.

  • X-Kaiterra-HMAC (string): Contains the base64-encoded HMAC (SHA256) produced from signing a certain subset of the request headers and payload with the secret key. The payload to sign consists of exactly the following items:

    • The X-Kaiterra-Client and X-Kaiterra-Time headers, represented as if they were URL query string parameters, using the header values exactly as they appear in the request
    • The URL and query string starting from immediately after the /v1 at the start of the URL (including the second /), exactly as it appears in the request
    • The request body, if any

For HMAC generation, these items must be represented as streams of bytes (header fields and URL are encoded as ASCII, and request body as UTF-8), and then directly concatenated together. This stream of bytes is what is then signed precisely according to the algorithm detailed on Wikipedia here. We use the SHA256 hash algorithm.

Conventions ¶

  • Laser Eggs are represented by 128-bit UUIDs. They’re case-insensitive and may be submitted with or without intermediate dashes (3d6d04a2-ba9f-11e6-9598-0800200c9a66 and c9f7cf2f98064884aa860315e85951f6 are both OK).

    • Note: there are no permissions on laser eggs, so anyone with access to the API can request data for any laser egg for which they know the ID. But Laser Egg IDs are 128 bits long, which makes them essentially impossible to guess.

  • All dates/times follow RFC3339 (a refinement of ISO8601), which looks like 2016-12-07T05:32:16Z. Milliseconds are accepted but are ignored (truncated). All times must be in UTC; any that aren’t will be rejected with 400 bad request.

  • All time ranges are labelled by the ending time, not the starting time.

Request Headers ¶

The following headers are accepted:

  • Accept-Encoding: gzip is supported; clients are encouraged to use it.

  • Content-Encoding: if there is a request body, this header denotes its encoding. Always use UTF-8.

Sensor Readings ¶

Data on various pollutants or other metrics (like temperature and humidity) are returned in JSON dictionaries. Metric names are standardized across all APIs. A fairly complete list of possible metrics returned includes:

  • pm25: PM2.5 in micrograms per cubic meter, post-calibration

  • pm10: PM10 in micrograms per cubic meter, post-calibration

  • humidity: relative humidity in % (0-100)

  • temp: temperature in Celsius

All pollutant quantities have default units of micrograms per cubic meter.

Laser Eggs ¶

Laser Eggs  ¶

Laser Egg Info
GET/lasereggs/{id}

Retrieves the Laser Egg’s most recent reading, which represents the average of the air sampled by the Laser Egg over the previous minute. As mentioned above in Conventions, the time stamp of the reading is the end of the minute-long interval over which the reading was taken.
